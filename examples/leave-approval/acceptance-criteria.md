# Leave Approval Workflow — Acceptance Criteria

> Acceptance criteria — generated by Product Copilot

## Leave request submission & balance validation

### [ ] AC-1 — Employee submits a request within available balance  _Happy path_

- **Given** Employee has sufficient leave balance for the requested dates
- **When** Employee submits a leave request
- **Then** Request enters the 'submitted' state
- **And** Request is routed to the manager for approval
- **And** Balance shows the request as pending/reserved

### [ ] AC-2 — Employee attempts to request more days than available  _Error_

- **Given** Employee's remaining balance is lower than the number of requested days
- **When** Employee attempts to submit the request
- **Then** The system blocks submission or flags the request as exceeding balance
- **And** A clear message states the balance shortfall
- **And** No approval routing occurs until resolved

### [ ] AC-3 — Overlapping requests are accepted at submission but re-checked at approval  _Edge case_

- **Given** Employee has two leave requests covering overlapping dates, both currently pending
- **When** An approver attempts to approve the second overlapping request
- **Then** The system re-validates balance and overlap at approval time
- **And** If the combined approval would breach balance or business rules, the approver is warned or the approval is blocked

## Multi-level approval circuit

### [ ] AC-4 — Manager approves a request under the escalation threshold  _Happy path_

- **Given** Request is submitted and within the manager's approval threshold
- **When** Manager approves the request
- **Then** Request moves to 'approved' state
- **And** Employee balance is decremented
- **And** Employee is notified of the approval

### [ ] AC-5 — Request above threshold requires HR validation after manager approval  _Happy path_

- **Given** Request exceeds the client-defined threshold requiring HR sign-off
- **When** Manager approves the request
- **Then** Request moves to a pending-HR state rather than fully approved
- **And** HR receives a notification to validate
- **And** Request only reaches 'approved' after HR validates

### [ ] AC-6 — Manager rejects a request  _Error_

- **Given** Request is pending manager approval
- **When** Manager rejects the request
- **Then** Request moves to 'rejected' state
- **And** Employee is notified with the rejection
- **And** No balance is deducted

### [ ] AC-7 — Approval circuit differs by client, country or contract  _Edge case_

- **Given** Two clients (or two countries) have different configured approval circuits
- **When** A request is submitted under each client's configuration
- **Then** Each request follows its own client-specific circuit (steps, thresholds, approvers)
- **And** No client's requests are routed using another client's rules

## Auto-escalation on approver inaction

### [ ] AC-8 — Request auto-escalates after 5 business days of manager inaction  _Happy path_

- **Given** Request has been pending with the manager for 5 business days with no action
- **When** The escalation timer elapses
- **Then** Request is automatically escalated to the next approver or HR
- **And** The new approver is notified
- **And** The escalation is recorded in the request history

### [ ] AC-9 — Manager approves at the same moment escalation fires  _Edge case_

- **Given** Escalation timer is about to elapse for a pending request
- **When** Manager submits an approval decision concurrently with the auto-escalation trigger
- **Then** Only one final outcome is recorded for that approval step
- **And** The employee and approvers do not receive duplicate or contradictory notifications
- **And** The audit trail shows a single, unambiguous resolution of the race

## Delegation of approval authority

### [ ] AC-10 — Delegate approves on behalf of an absent manager  _Happy path_

- **Given** Manager has set up a delegation to another approver
- **When** The delegate approves or rejects a request routed to the manager
- **Then** The decision is applied to the request as if made by the manager's approval step
- **And** The request progresses through the workflow normally

### [ ] AC-11 — Delegate actions are traceable in the audit trail  _Permission_

- **Given** A delegate has acted on a request on behalf of a manager
- **When** Someone reviews the request's history or audit trail
- **Then** The audit trail clearly identifies the delegate as the actor, distinct from the original manager

## Cancellation of leave

### [ ] AC-12 — Employee requests cancellation of an approved future leave  _Happy path_

- **Given** Leave request is in 'approved' state and has not started
- **When** Employee requests cancellation
- **Then** Cancellation request is routed to HR for validation
- **And** Leave remains active until HR validates the cancellation

### [ ] AC-13 — Cancellation of leave that has already started  _Edge case_

- **Given** Leave request is 'approved' and the leave period has already begun
- **When** Employee or manager requests cancellation of the started leave
- **Then** Cancellation still requires HR validation before taking effect
- **And** Balance is adjusted correctly for only the unused remaining days once validated

### [ ] AC-14 — Manager cannot unilaterally cancel an approved request  _Permission_

- **Given** Leave request is approved
- **When** Manager attempts to cancel the request directly without HR validation
- **Then** The system blocks the cancellation or requires HR sign-off before it takes effect
- **And** The attempt is visible in the audit trail

## Notifications & calendar sync

### [ ] AC-15 — Notifications sent on key workflow events  _Happy path_

- **Given** A leave request changes state (submitted, approved, rejected, escalated, cancelled)
- **When** The state change occurs
- **Then** Relevant actors (employee, approver, HR) receive an email and in-app notification
- **And** Notification content matches the event that triggered it

### [ ] AC-16 — Approved leave is pushed to the calendar  _Happy path_

- **Given** A leave request reaches 'approved' state
- **When** The approval is finalised
- **Then** The leave appears in the connected calendar
- **And** Changes made directly in the external calendar do not modify the leave request in the system

## Permission boundaries

### [ ] AC-17 — Only HR can edit an approved request  _Permission_

- **Given** A leave request is in 'approved' state
- **When** A manager or employee attempts to edit the request
- **Then** The edit is blocked for manager and employee
- **And** HR is able to perform the edit

### [ ] AC-18 — Employee cannot approve their own request  _Permission_

- **Given** Employee has submitted a leave request
- **When** The same employee attempts to approve or validate their own request
- **Then** The action is blocked
- **And** Only a designated approver or HR can approve the request

### [ ] AC-19 — Administrator can configure the approval circuit per client  _Permission_

- **Given** Administrator has access to circuit configuration for a given client
- **When** Administrator changes the approval circuit (steps, thresholds, approvers)
- **Then** The change applies only to that client's future requests
- **And** Other clients' configurations remain unaffected
- **And** The configuration change is recorded in an audit trail

## Multi-tenant isolation & pilot rollout

### [ ] AC-20 — Pilot client configuration does not affect non-pilot clients  _Edge case_

- **Given** The workflow is enabled for a pilot client only
- **When** Leave requests are submitted by employees of both pilot and non-pilot clients
- **Then** Pilot client employees use the new digital workflow
- **And** Non-pilot clients continue on their existing process without disruption
